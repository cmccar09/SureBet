Master Flow

Daily Orchestration Timeline

08:30 — Scheduled Task: SureBet-Morning-Racecard
Phase 1 · SL Racecard Ingestion
sl_racecard_fetcher.py fetches today's full UK racecard from Sporting Life. Every race gets: trainer, jockey, weight, form figures, official rating, Timeform stars, Rating123, draw, pre-race odds, last-6-run history, going, class, distance, tipster verdict text. Saved to racecard_cache.json and DynamoDB SureBetRacecards.
09:00 — Scheduled Task: SureBet-Daily-Health-Check
Phase 2 · Pre-Flight Health Check
daily_health_check.py validates that all required data files exist and are fresh, DynamoDB tables are accessible, Race data has form, Betfair credentials are valid. Aborts the pipeline if critical issues found; emits warnings for non-blockers.
09:30 — Scheduled Task: SureBet-Betfair-Fetch
Phase 3 · Betfair Live Data Ingestion
betfair_odds_fetcher.py authenticates via AWS Secrets Manager and fetches all UK WIN markets for the next 6 hours from the Betfair Exchange API. Returns live BSP odds, form strings, trainer, jockey, draw, last-run info per runner. Saved to response_horses.json. Also enriched with form_enricher.py (Racing Post last-6-race history: course, distance, going, position, class).
10:00 — Scheduled Task: SureBet-Comprehensive-Analysis
Phase 4 · Full Field Analysis (100% Coverage)
comprehensive_workflow.py loops every race and every runner. Builds meeting-level context (trainer/jockey focus signals, new-trainer flags). Scores every horse through the 40-signal scoring engine in comprehensive_pick_logic.py. Picks passing 4-tier validation are written to DynamoDB SureBetBets with show_in_ui=True. All horses scored (not just picks) for learning. Mandatory 90% field coverage — races below threshold are skipped.
10:30 — Automated
Phase 5 · Top-3 Gap Validation & UI Publication
After scoring, only picks with a score gap ≥ 20pts over the next-highest horse are flagged for UI. The Lambda function BettingPicksAPI serves validated picks via API Gateway to the React frontend on AWS Amplify. Weak-gap picks are suppressed (show_in_ui=False, rejection reason stored).
Every 30 minutes (10:00 – 21:00) — Task: SureBet-Hourly-ResultsFetcher
Phase 6 · Live Result Capture
sl_results_fetcher.py runs every 30 minutes. Primary: hits sportinglife.com/racing/fast-results — single JSON fetch, 30 most recent results globally, instant after weigh-in. Fallback: per-race HTML page scraping for any race not yet in the fast feed. Records: outcome (win/placed/loss), finish position, profit/loss, winner name, winner trainer/jockey, tote win/place, exacta, tricast, winning margins, non-runner count. Also pulls pick's own racecard data: trainer, form, Timeform stars.
Every 30 minutes (10:00 – 21:00) — Continuous
Phase 7 · Coordinated Learning Loop
coordinated_learning_workflow.py runs a background loop: analyzes ALL horses (learning track), matches results as they arrive, updates signal weights in DynamoDB based on what's winning. Dynamic weights are hot-loaded (5-min cache) on every scoring call — no restart needed.
20:00 — Scheduled Task: SureBet-Evening-Loss-Report
Phase 8 · Evening Loss Analysis & Report
evening_loss_report.py queries today's settled picks, identifies red flags on each loss (extreme odds, going mismatch, class mismatch, low coverage, trainer absence, score gap too narrow), generates an HTML report saved to reports/, and emails it to charles.mccarthy@gmail.com via AWS SES.
21:00 — Automated
Phase 9 · Learning Cycle & Weight Update
daily_learning_cycle.py loads the last 7 days of settled picks from DynamoDB, runs learning_engine.analyze_performance_patterns() to detect overweight signals, calls update_prompt_with_learnings() to persist updated weights back to DynamoDB. Weights take effect immediately (hot-loaded cache) — the model improves overnight ready for the next day's analysis run.

Data Sources & Ingestion

  • Fetches full UK/Irish race card from /racing/racecards/{date}
  • Per race: going, distance, class, age-band, handicap flag, tipster verdict
  • Per runner: trainer, jockey, weight, draw, official rating, age/sex

  • Per runner: last 6 runs (course, distance, going, position, class, odds)
  • Output: racecard_cache.json · SureBetRacecards DynamoDB
📊 Betfair Exchange API 09:30 daily

Script: betfair_odds_fetcher.py

  • Authenticates via AWS Secrets Manager (session token)
  • Fetches all UK WIN markets for the next 6 hours
  • Returns: BSP odds, form, trainer, jockey, draw, last-run, weight carried
  • All UK horse racing markets (Event Type 7)
  • Output: response_horses.json — live BSP odds per runner
📜 Racing Post Form Enricher On demand

Script: form_enricher.py

  • Scrapes Racing Post profiles for last-6-race history per runner
  • Per run: date, course, distance (furlongs), going, position, field size, OR, class
  • 12-hour local cache in form_cache.json — avoids repeat scraping
  • Powers: exact course win (+20), exact distance win (+20), going win (+16), close 2nd (+14), fresh window (+10), OR rising (+10), big-field win (+8)
🌦 Weather & Going Inference Continuous

Module: weather_going_inference.py

  • Fetches live going reports per track for race day
  • Infers going (Good, Soft, Heavy, AW Standard) from SL data
  • Used by scoring: going_suitability (+16), heavy_going_penalty (−12)
  • 1-hour cache to avoid excessive fetches
⚡ SL Fast-Results Feed Every 30 min

Source: sportinglife.com/racing/fast-results

  • Single-page JSON feed (__NEXT_DATA__), 30–40 most recent races globally
  • Available within seconds of weigh-in
  • Contains: winner + top-3, draw, SP, favourite flag, race details
  • Also captures: tote win/place, exacta, tricast, winning distances, runner count
  • Fallback: full per-race HTML scrape for races not yet in feed
🗄 DynamoDB Tables eu-west-1
  • SureBetBets — picks, scores, results, P&L (PK: bet_date)
  • SureBetRacecards — pre-race racecard data per race
  • SureBetWeights — dynamic scoring signal weights
  • RacingPostRaces — enriched form history cache

✅ Pick Approval Criteria
  • Score above minimum threshold for grade tier
  • Score gap >20pts over next-highest horse in field
  • Race not in skip list (e.g. Class 3–6 handicaps too unpredictable)
  • Field coverage ≥90% (minimum 90% of runners must be scored)
  • Odds not extreme (<20.0 BSP preferred)
  • 100% field analysis mandatory — no partial-field picks
❌ Automatic Reject Conditions
  • Score gap <20pts — model cannot separate with confidence
  • Race coverage <90% — incomplete field data
  • Horse flagged as AW Class 5/6 (−50pt penalty applied)
  • Heavy going with no proven form on the ground
  • Race type in should_skip_race() blocklist
  • No Betfair market data available (no runners object)

P&L Calculation Logic
  • WIN: profit = (BSP odds − 1) × stake
  • PLACED (2nd/3rd): −stake (win-only bet)
  • LOSS/Unplaced: −stake
  • Stake scaled by grade tier (×2.0 / ×1.5 / ×1.0 / ×0.5)
  • Base stake configured per bet (default £6)
Match Algorithm
  • Normalise course name to lowercase
  • Match race time ±15 minute window
  • Normalise horse name: strip (IRE)/(FR) suffixes, lowercase
  • Find finishing position by index in ordered list
  • If horse not in top 3 list → infer Unplaced/LOSS

Phase 8

Evening Loss Analysis

evening_loss_report.py — 20:00 daily via BettingLossReport task

For every loss today, the system applies the red-flag detector:

  • 🔴 Odds > 20/1 — extreme outsider, model blind spot
  • 🔴 Going mismatch — won on opposite going type
  • 🔴 Score gap <10pts — model unable to separate field
  • 🔴 Field coverage <70% — incomplete analysis
  • 🔴 Race class mismatch — stepped up in class unexpectedly
  • 🔴 Trainer/jockey absent from our data
  • 🔴 Score inflated by going bonus alone (>35% of total)
  • 🔴 Winner was lower-scoring horse — learning opportunity

Report saved to reports/{date}_loss_report.html and emailed to charles.mccarthy@gmail.com via AWS SES. Archive of all daily reports kept in reports/ folder.

⚡ Dynamic Weight Hot-Loading
get_dynamic_weights() in comprehensive_pick_logic.py loads from DynamoDB with a 5-minute cache. No service restart needed — updated overnight weights are live by 09:00 analysis run.
📐 Pattern Detection
learning_engine.analyze_performance_patterns() looks at the last 7 days of bets, identifies which signals appeared in winners vs. losers, and adjusts weights accordingly. Score inflation from single signals is detected and penalised (e.g. going bonus cap).
🛡 Score Gap Enforcement
Known lesson: Fahrenheit Seven had +1pt gap and was immediately removed. The minimum score gap (currently 20pts) acts as a confidence threshold — a model that can't decisively separate horses doesn't bet. Gap is tracked and can be auto-raised by the learning system if win rate drops.
🧩 Dual-Track Analysis
Every horse in every race is scored (show_in_ui=False) so the learning system sees the full field distribution, not just our picks. This reveals which signals predicted the actual winner even when we didn't back it.
🔁 Coordinated Loop
coordinated_learning_workflow.py runs an intra-day loop: analyse races → wait for results → learn from results → apply to next race predictions. Weights can shift during the race day based on morning results feeding into afternoon picks.
📝 Prompt & Insight Archive
update_prompt_with_learnings() writes human-readable summaries to prompt.txt and daily insights via track_daily_insights.py. This creates a growing knowledge base of lessons-learned that feed future analyses.

Resilience

Self-Healing & Fault Tolerance

✅ Daily Health Check
daily_health_check.py at 09:00 validates data files, DynamoDB access, Betfair credentials, form data presence. Blocks the pipeline on critical failures. Non-blockers emit warnings only.
🔄 Fast-Results Fallback
If the fast-results JSON feed is unavailable or doesn't contain a race, the system automatically falls back to scraping individual per-race HTML pages. No human action required; fallback triggered per unresolved pick.
⚖ Weight Fallback
If DynamoDB SureBetWeights is unreachable, get_dynamic_weights() returns the hardcoded DEFAULT_WEIGHTS dict in comprehensive_pick_logic.py. Analysis always completes.
📦 Racecard Cache Fallback
sl_racecard_fetcher.py is optional — if import fails, sl_results_fetcher.py sets _RACECARD_AVAILABLE = False and continues without trainer/jockey enrichment. Core result capture is unaffected.
🛑 Coverage Guard
Any race where <90% of runners can be scored is skipped entirely. Lesson from Carlisle 15:35 (9% coverage → system failure). Better no pick than a bad pick formed without full-field data.
🔒 AW Class 5/6 Guard
Lesson from Beauzon (91pts, finished 6th, AW Class 5): a −50pt penalty is applied to AW Class 5/6 entries. This effectively blocks them from reaching the pick threshold regardless of form streak.
