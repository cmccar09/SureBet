⚙️ Daily Pipeline — How Picks Are Made Each Day

complete_daily_analysis.py runs once per day (triggered at 12:00, 14:00, 16:00, 18:00 by a Lambda cron). Below is every stage it executes in order before a pick is published to the UI. The red stage was missing during all 61 losses in this report.

#StageWhat it doesFailure modeStatus during losses
0 Betfair odds fetch
betfair_odds_fetcher.py		 Calls Betfair API for every UK/IRE race today. Writes response_horses.json with runners, SP odds, draw, weight, going, trainer, jockey. Also stores Betfair market IDs for settlement matching later. If Betfair token is stale the file will be empty — nothing is scored and no picks are generated. ✓ Running (runs before complete_daily_analysis.py)
1 Deep form enrichment
form_enricher.enrich_runners()		 Scrapes Racing Post / Sporting Life for each runner's last-6-race history. Injects form_runs onto every runner dict. The scoring engine then fires: exact_course_win (+20), exact_distance_win (+20), going_win_match (+32), close_2nd_last_time (+14), fresh_days_optimal (+10), or_trajectory (+10), big_field_win (+8). Combined ceiling: +114 pts. Without this step, all seven signals score 0 for every horse. The model still runs but is blind to recent form quality. ✗ NOT CALLED during all 61 losses
module existed but was never imported or invoked.
Fixed 2026-03-30 — now wired as Stage 1/5.
2 Horse history load
DynamoDB SureBetBets scan		 Scans DynamoDB for all settled bets (result_won set) to build a win-rate map per horse. Applies a DB history bonus: >40% win rate → +15 pts, >25% → +8 pts, >10% → +4 pts. Only meaningful once 100+ horses have results recorded. Cold start: first few weeks there is no history, bonus is always 0. Not a bug — degrades gracefully. ✓ Running (no data in early period, but correctly handled)
3 Comprehensive scoring
comprehensive_pick_logic.analyze_horse_comprehensive()		 7-factor model: jockey_quality, trainer_reputation, going_suitability (capped 10 pts), consistency (6 pts weight), optimal_odds, distance_suitability, cd_bonus, plus market_leader overlay (fav +16, co-fav +10, 2nd choice +5) and deep_form (from Stage 1). Going suitability was previously uncapped (up to 30+ pts) which bloated scores for unsuited horses. Capped at 10 from 2026-03-30. ✓ Running
4 Quality gates (S1–S6)
_passes_quality_gates()		 Six filters applied before a race-best can become a UI pick: S1 non-market picks need ≥85 pts; S2 requires at least one anchor signal (market_leader / trainer / cd_bonus / unexposed); S3 age-padded picks without market backing need ≥92; S4 score gap vs 2nd-best must be ≥8 pts; S5 Saturday tightening (gap ≥15, large-field needs market); S6 16+ runner fields without market agreement need gap ≥12. Over-tightening gates (raised threshold to 85 + halved market_leader bonus in Mar 2026) produced 0 picks for 2 straight days. Restored market_leader bonus and set threshold to 78 on 2026-03-27. ✓ Running (6 gates active)
5 Pick selection & save
DynamoDB put_item + manifest		 Best-per-race sorted by score, top 3 with show_in_ui=True saved to DynamoDB. All other horses saved as learning records. Analysis manifest (pipeline status + signal coverage %) written to SYSTEM_ANALYSIS_MANIFEST / STATUS record — the UI reads this to show the pipeline checklist banner. If fewer than 3 races pass gates, fewer than 3 picks are shown. No picks is a valid outcome (model is uncertain — do not force a bet). ✓ Running
