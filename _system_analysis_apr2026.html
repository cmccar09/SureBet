Root Cause Analysis

1. Short-Price Picks: Structurally Negative EV

The system's observed overall win rate is 33.7%. Even the "best" picks probably achieve ~40–45% win rate. For a bet to have positive expected value, the win probability must exceed the implied odds probability:

2. Score ≠ Win Rate — The Calibration Gap

Multiple days show scores 90–123 with 0 wins. The model score correctly ranks horses within a race but the absolute value of the score is not a reliable predictor of win probability across races. A horse scoring 118 at 1.53 odds has no better chance of winning than the market implies (≈65% chance), and our model can't consistently beat that.

3. The Odds Sweet Spot: Historical Evidence

Losing Bands (Historical)

  • 2/1–4/1 (3.0–5.0 decimal): -£11.95 P&L
  • Below 6/4 (below 2.5): structurally -EV

Winning Bands (Historical)

  • 4/1–7/1 (5.0–8.0 decimal): +£25.20 P&L
  • 8/1–9/1 (9.0+): partial — needs backing from model

4. Race Type Variance

Large-field handicaps (16+ runners) are uniquely unpredictable — the handicapper designs them to be competitive. Gate S6 already rejects large-field without market backing unless score_gap ≥ 12. However, large-field picks with market backing can still get through and lose at higher rates than smaller field races.

5. Price Movement Signal Under-Used

The scoring code has a price steam/drift system (price_movement field), but this only fires when betfair_odds_fetcher.py provides price movement data between two snapshots. This is not consistently captured. Steam (price shortening ≥20%) should be one of the strongest buy signals: it means smart money is entering the market.

Priority Improvements (Not Yet Implemented)

1
Betfair Price Movement: Capture Two-Snapshot Steam/Drift HIGH IMPACT

The scoring engine already has a price steam handler (lines 297–314 in complete_daily_analysis.py), but betfair_odds_fetcher.py only captures a single snapshot. Steam (price shortening ≥20%) is the single most reliable external signal in horse racing markets — it means professional money is entering.

What to build:

  • In betfair_odds_fetcher.py: on the Refresh run (90 mins before first race), compare current Betfair SP to the morning snapshot stored in DynamoDB.
  • Calculate price_move_pct = (morning_odds - current_odds) / morning_odds * 100
  • Attach price_movement='steaming' if shortened ≥20%, or 'drifting' if extended ≥25%
  • The scoring engine will then auto-apply +12–15pts for steam, -8pts for drift
Expected impact: +5–8% win rate on steam picks; flags drift-backed long-shots before they're selected
2
Race Type Score Normalisation HIGH IMPACT

Currently all races get the same scoring thresholds regardless of race class or type. A Class 2 conditions race (8 runners, small field, less variance) and a Class 4 handicap (14 runners, more variance) both need score ≥ 78 to be eligible. But the probability of correctly picking the winner is very different. Score thresholds should be adjusted upward for handicaps.

Suggested implementation:

  • Non-handicap conditions/maiden races: MIN_CONFIDENCE = 75 (lower threshold OK)
  • Handicap races: MIN_CONFIDENCE = 85 (require higher conviction)
  • Class 5/6 handicaps: blocked unless score ≥ 90 AND market-backed (existing S6 + low-class penalty)
Expected impact: Reduces handicap losses, focuses picks on more predictable race types
3
Calibrated Win Probability vs Odds Check MEDIUM IMPACT

Add a sanity check: if the model scores a horse 90+ but the market offers it at < 3.0 (implied probability > 33%), the "market" and "model" agree the horse is likely to win — but at 3.0 odds, even if we're right 40% of the time, it's marginal EV. Consider a flag:

  • If score ≥ 90 AND odds < 3.0: add warning label "Short-price favourite — high confidence but limited EV"
  • Surface this warning clearly on the WhatsApp notification and UI
Expected impact: Helps human review when backing short-price picks; improves decision-making
4
Database History: Race Type Matching MEDIUM IMPACT

Currently the 15pts "database history" bonus fires for any past win by the horse in our system, regardless of race type. A horse with a National Hunt Flat win shouldn't boost when it's running in a 2m Chase on heavy ground.

Suggested implementation:

  • When loading DB history for a horse, filter by race_type SIMILAR TO today's race type
  • Full bonus (15pts) only for wins in same race type + similar distance (±2f)
  • Half bonus (7pts) for wins in same race type but different distance
  • Quarter bonus (4pts) for wins across race types (different discipline)
Estimated effort: 2–3 hours. Expected impact: More accurate form signals for cross-discipline horses
5
Going Confidence Gate: Official vs Estimated Ground LOW-MEDIUM

The going system reads official declarations (use_official=True), but early-morning going is often listed as "Good to Soft" pending inspection. A horse rated as "suited to soft" may be selected based on presumed conditions that change by post time.

Suggested implementation:

  • When going is marked "under examination" or "awaiting inspection": reduce going suitability score by 50%
  • In the WhatsApp notification, flag picks as "⚠️ GOING AWAITING INSPECTION"
  • Don't reject outright, but reduce confidence so other factors must compensate
Estimated effort: 1 hour. Prevents over-rewarding going suitability on assumed conditions
6
Draw Bias Integration for Flat Turf LOWER PRIORITY

For flat turf races, draw position (stall number vs rail) is a well-established signal. Certain courses strongly favour low draws at short trips, others favour high draws. Currently the model has no draw awareness.

What this requires:

  • Betfair course data to include stall/draw numbers (varies by feed)
  • Historical draw statistics by course + distance (e.g. Lingfield 5f: low draw +8%)
  • Data source: could be supplemented by Timeform or Racing Post draw stats
Estimated effort: 4–6 hours. Medium impact specifically on short-trip flat racing

System Architecture (Current State)

Pick Selection Flow
  1. Morning fetch: sf_betfair_fetch → betfair_odds_fetcher.py — pulls live Betfair odds for all today's races, stores to S3
  2. Morning analysis: sf_analysis → complete_daily_analysis.py — scores ALL horses using comprehensive_pick_logic.py (30+ factors)
  3. Selection: Top-scored horse per race → quality gates S1–S9 → diversity tier selection → TARGET_PICKS=3 picks marked show_in_ui=True
  4. Notification: sf_notify sends WhatsApp with picks ranked by confidence
  5. Refresh: 90 mins before races, re-runs analysis with live odds (intraday picks can be added, max 2)
  6. Results: Evening step function fetches results, marks WIN/LOSS/PLACED
  7. Learning: SureBet-Learning adjusts weights in DynamoDB based on settled picks

What the System Needs to Hit 2+/3 Target

Realistic path: The 2/3 target (66.7% win rate) is ambitious. A more achievable short-term target is 45% pure win rate on the 3 morning picks — which corresponds to winning 1.35 per day on average. 2+ win days become regular (roughly 3 out of 5 days) with 2+ wins. With the S9 gate now blocking sub-2.5 picks, the effective picks are concentrated in higher-value odds bands.

Key Actions in Priority Order:

  1. ✅ S9 minimum odds gate (2.5+) — DONE. Eliminates systematic negative-EV short-price picks.
  2. ✅ API results upper bound fix — DONE. Results page now shows accurate historical data including high-score losses.
  3. 💡 Betfair two-snapshot price movement — The scoring engine already handles it. Just need betfair_odds_fetcher.py to compare morning vs refresh odds and attach price_movement field. Steam picks are significantly more reliable.

  5. 💡 Going confidence gate — When going is "under inspection," halve the going_suitability bonus. Prevents over-confidence on assumed condition picks.
