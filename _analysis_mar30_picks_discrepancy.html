How to Fix This

  • 1 Prevent double workflow runs on the same day. Add a DynamoDB guard record (e.g. bet_date="2026-03-30", bet_id="WORKFLOW_RUN_LOCK") at the start of the workflow and abort if it already exists. Only clear it the following day.
  • 2 Never overwrite show_in_ui=True picks on re-run. Before updating a DynamoDB record, check if show_in_ui=True already. If so, preserve the existing pick rather than re-scoring it — or at minimum write to a new versioned record rather than mutating the existing one.
  • 3 Fix the bet_id timestamp format inconsistency. The third run wrote 2026-03-30T14:18:00+00:00_… (with colons) while all other runs use 2026-03-30T141800+0000_…. This means Run 3 created NEW records instead of overwriting Run 2's, escaping the lock. Normalise the bet_id format in one place.
  • 4 Log workflow run start/end timestamps. Write a WORKFLOW_RUN_LOG record each time the workflow triggers so you can audit when re-runs happened and what changed.
  • 5 Investigate Your Darling score drop (96 → 73). A 23-point swing suggests a going or odds update was applied mid-day that significantly penalised it. That horse WON — worth understanding what signal contradicted the market.
